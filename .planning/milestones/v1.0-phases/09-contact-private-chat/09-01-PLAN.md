  <action>
Create the migration exactly per 09-RESEARCH.md Pattern 1 (the SQL there is the design — reproduce it with full policies):

1. **contact_log**: identity PK, listing_id/buyer_id/seller_id FKs, buyer_name/buyer_email/buyer_phone/message_text, admin_emailed_at, created_at. RLS: buyer INSERT (`(select auth.uid()) = buyer_id`), buyer SELECT own rows, seller SELECT rows where `(select auth.uid()) = seller_id`. NO update/delete policies (immutable log — MSG-04). Index on (buyer_id, created_at) for the daily rate-limit head-count.
2. **message_threads**: listing_id/buyer_id/seller_id/contact_log_id (NOT NULL — makes invariant #5 structural), created_at, last_message_at, buyer_last_read_at/seller_last_read_at (read watermarks), buyer_emailed_at/seller_emailed_at (email-throttle watermarks), buyer_hidden_at/seller_hidden_at (hide ≠ delete), `unique (listing_id, buyer_id)`. RLS: SELECT + UPDATE for participants (`(select auth.uid()) in (buyer_id, seller_id)`); INSERT buyer-only (`(select auth.uid()) = buyer_id`). Indexes on (buyer_id), (seller_id), (last_message_at desc).
3. **messages**: identity PK, thread_id FK, sender_id FK, `body text not null check (char_length(body) between 1 and 2000)`, created_at. Index `(thread_id, created_at)`. RLS SELECT: ONE indexed EXISTS against message_threads PK — `exists (select 1 from public.message_threads t where t.id = messages.thread_id and (select auth.uid()) in (t.buyer_id, t.seller_id))` (this runs per-change × per-subscriber under WALRUS — keep it exactly this cheap; do NOT add the block check here, blocks gate sending not reading). RLS INSERT: sender self-attribution + participant + NOT blocked either direction via NOT EXISTS on user_blocks (Research Pattern 1 SQL). NO update/delete policies. **Qualify the NEW row's columns as `messages.thread_id` inside policy subqueries** (0015 depth-1 bug lesson).
4. **user_blocks**: composite PK (blocker_id, blocked_id), `check (blocker_id <> blocked_id)`. RLS: owner INSERT/DELETE/SELECT (`(select auth.uid()) = blocker_id`).
5. **reports**: reporter_id, exactly-one-of listing_id / comment_id / message_id (`check (num_nonnulls(listing_id, comment_id, message_id) = 1)` — fitment_rules precedent), `comment_id ... on delete cascade` (comments hard-delete; the deletion audit table keeps the trace — note this in a SQL comment), reason CHECK in ('scam_fraud','harassment','spam','prohibited_item','wrong_info','other'), optional detail, created_at. Three partial unique indexes (reporter_id, target) for one-report-per-user-per-item. RLS INSERT: `(select auth.uid()) = reporter_id` AND, when message_id is not null, reporter must be a participant of that message's thread (EXISTS messages→message_threads — otherwise message ids are guessable); reporter SELECT own rows. NO update/delete.
6. `alter table public.profiles_private add column message_email_opt_out boolean not null default false;`
7. `alter publication supabase_realtime add table public.messages;` — REQUIRED, Pitfall 1.

Every table: `enable row level security` in this same migration (invariant #2). Apply to Staging: `npx supabase db query --linked -f supabase/migrations/0016_messaging.sql`, then verify live: all 5 tables have `relrowsecurity = true` in pg_class, and `select * from pg_publication_tables where pubname = 'supabase_realtime' and tablename = 'messages'` returns 1 row.
