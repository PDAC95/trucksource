```ts
export type CommentItem = {
  id: number; parentId: number | null; body: string; createdAt: string;
  authorId: string;           // needed by the UI to show "delete" on own comments

```
Implementation mirrors lib/search/queries.ts batch hydration (Pitfall 5 — no N+1):
1. ONE enumerated select: `from("listing_comments").select("id, listing_id, author_id, parent_id, body, created_at").eq("listing_id", listingId)` — NEVER `select('*')`, NEVER any profiles embed.
2. ONE batched `from("profiles_public").select("id, username, display_name").in("id", uniqueAuthorIds)`. Authors whose profile is missing fall back to "usuario" — never crash.
3. Bucket in app code: parents (`parent_id === null`) sorted `created_at DESC` (newest first, LOCKED); replies grouped under their parent sorted `created_at ASC` (conversation order, LOCKED). Orphan replies (parent missing mid-read) are dropped.
Return `[]` on error. Anon-callable (comments are public-read) — use the cookie server client like getListing does.

Mirror the exact column lists asserted by `COMMENT_SELECT_COLUMNS` / `COMMENT_AUTHOR_COLUMNS` in tests/integration/social.contract.test.ts (08-01) — keep them in lockstep.

```ts
export type AddCommentResult = { ok: true; id: number } | { ok: false; error: "unauthenticated" | "invalid" | "rate_limited" | "comments_closed" | "not_found" };
export async function addComment(input: unknown): Promise<AddCommentResult>;
export type DeleteCommentResult = { ok: true } | { ok: false; error: "unauthenticated" | "invalid" | "not_found" };
export async function deleteComment(id: number): Promise<DeleteCommentResult>;
export type MarkCommentsSeenResult = { ok: true } | { ok: false; error: "unauthenticated" | "invalid" | "not_found" };
export async function markCommentsSeen(listingId: number): Promise<MarkCommentsSeenResult>;
```

**addComment** guard ORDER (the unit test asserts it): (1) getClaims → unauthenticated short-circuit; (2) `commentSchema.safeParse` → invalid; (3) rate limit: count OWN comments in the last 60s (`from("listing_comments").select("id", { count: "exact", head: true }).eq("author_id", userId).gte("created_at", new Date(Date.now()-60_000).toISOString())`) → over 5 returns `rate_limited` (discretion: generous, dependency-free — Research "Don't Hand-Roll"); (4) pre-check the listing is active+unexpired (enumerated read of `id, status, expires_at`) → sold/expired returns `comments_closed`, missing returns `not_found` — clean message; RLS remains the real enforcement; (5) insert `{ listing_id, author_id: userId, parent_id: parsed.parentId ?? null, body }` with `.select("id")`; an RLS rejection here (e.g. crafted depth-2 parentId) collapses to `invalid`. Then `revalidatePath(\`/listings/\${listingId}\`)`.

**deleteComment:** validate id (positive int) → `.from("listing_comments").delete().eq("id", id).select("id, listing_id")`. RLS decides authorization (author OR seller) — do NOT re-implement the seller check; zero rows → `not_found` (no existence leak). On success `revalidatePath(\`/listings/\${listing_id}\`)` using the returned row. Cascade + audit happen in the DB.

**markCommentsSeen:** owner-scoped `.from("listings").update({ comments_seen_at: new Date().toISOString() }).eq("id", listingId).eq("seller_id", userId).select("id")`; zero rows → `not_found`. No revalidatePath needed (/sell/listings is force-dynamic).
