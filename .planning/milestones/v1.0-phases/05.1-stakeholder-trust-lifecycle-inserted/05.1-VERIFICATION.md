### Cross-Cutting Privacy Gate

The privacy/RLS gate was re-verified against 0010:

- `seller_type` and `display_name` are additive columns on `profiles_public` (world-readable, owner-writable via the 0001 owner-update policy). No new RLS policy was added.
- No code path in `lib/actions/account.ts` reads `profiles_private`. The comment and code both confirm this.
- `updateDisplayName` returns `publicName` derived from `resolvePublicName(row.display_name, row.username)` — it selects `id, username, display_name` from `profiles_public` only.
- `app/(public)/u/[username]/page.tsx` selects `display_name, seller_type` from `profiles_public` only — never `profiles_private`, never `*`.
- `tests/integration/stakeholder-lifecycle.test.ts` gate (a) and (a'') structurally prove: anon reads of these columns contain none of `PII_KEYS`; anon updates change zero rows.
- `expires_at` and `status` on the `listings` table add no PII.
- The `notifications` table has owner-only SELECT (RLS) and no INSERT policy for authenticated users — only the service-role cron route writes rows.
