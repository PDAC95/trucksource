# Phase 05.1 Plan 05: Listing Lifecycle Automation Summary

Completed LIST-09's automation half: a DB-side daily expiry flip (`pg_cron`, migration 0011), a secret-guarded Vercel Cron route that emails sellers + writes an in-app notification ~7 days before expiry, and the owner-scoped notifications read surface. All code, the migration file, the cron config, the env doc, and the route-guard test are authored, committed, and green (tsc + build + full suite). **One infrastructure gate remains:** `pg_cron` is NOT yet enabled on the Staging Supabase project (the `cron` schema does not exist), so migration 0011 could not be applied/scheduled — this is a dashboard-only toggle the user must flip, then re-apply 0011 (see Deferred / Checkpoint below).

## What Shipped

**Task 1 — pg_cron flip migration + admin client (commit `5077a10`):**
- `supabase/migrations/0011_expire_listings_cron.sql`: idempotent (unschedule the same-named job if present, then `cron.schedule`) daily job `'expire-listings'` at `5 0 * * *` running `update public.listings set status='expired' where status='active' and expires_at <= now()` — ONLY active-past-expiry flips; `sold` (terminal) and active-future are untouched (Pitfall 3, gate c; encodes `shouldExpire`).
- `lib/supabase/admin.ts`: **already existed** from Phase 2 (server-only `createAdminClient()` over `SUPABASE_SERVICE_ROLE_KEY`) — reused as-is, not recreated.
- Apply to Staging FAILED: `ERROR 3F000: schema "cron" does not exist` → pg_cron is not enabled. The migration file is authored and committed; scheduling waits on the dashboard toggle.

**Task 2 — secret-guarded route + notifications reader (commit `aea05cc`):**
- `app/api/cron/near-expiry/route.ts` (`GET`, `force-dynamic`): GUARD FIRST (Pitfall 6) — rejects with 401 unless `Authorization === Bearer ${CRON_SECRET}` and CRON_SECRET is set, before ANY work. On auth, via the service-role admin client: selects active listings in the next ~7 days (`status='active'`, `expires_at` in `(now, now+7d]`), re-filters with `isExpiringSoon`, drops listings already having a `kind='listing_expiring'` notification (notify-once JS anti-join), resolves seller emails from `profiles_private` (the one legitimate server-only PII read, never returned), sends a best-effort Resend email (failure swallowed), and inserts a `kind='listing_expiring'` notification row per listing. Returns `{ notified: N }`.
- `lib/notifications/queries.ts`: `listMyNotifications()` + `unreadNotificationCount()` — owner-scoped via the COOKIE client (getClaims sub; the owner SELECT RLS policy scopes rows). NO service-role here (reads are owner-scoped). Returns `[]`/`0` when unauthenticated.

**Task 3 — vercel cron config + env doc + route-guard test (commit `741c238`):**
- `vercel.json`: new `crons` entry → `/api/cron/near-expiry` daily at `0 1 * * *` (after the 00:05 UTC DB flip).
- `.env.example`: documented server-only `CRON_SECRET` (Bearer guard; Vercel sends it automatically when set). `SUPABASE_SERVICE_ROLE_KEY` + `RESEND_API_KEY` were already documented from Phase 1/2.
- `tests/unit/near-expiry-cron.test.ts` (6 tests, mocked admin client + Resend): no-secret and wrong-secret → 401 with NO insert/email (Pitfall-6 guard); near-expiry-only + notify-once selection (excludes already-notified + out-of-window via the `isExpiringSoon` re-check); all-already-notified → `notified:0`; a Resend `fetch` failure still inserts the notification (best-effort email).

## CHECKPOINT — pg_cron enablement (human-action)

To finish scheduling the daily flip, the user must:
1. **Supabase Dashboard → (Staging project) → Database → Extensions → enable `pg_cron`.**
2. Re-apply the migration (idempotent):
   `supabase db query --linked -f supabase/migrations/0011_expire_listings_cron.sql`
3. Verify: `select jobname, schedule, command from cron.job where jobname='expire-listings';` returns the row.
4. **Vercel → Project (trucksource) → Settings → Environment Variables → add `CRON_SECRET`** (a random secret, all environments) so the deployed `/api/cron/near-expiry` cron is authorized.

Until step 1-3, listings will not auto-flip to `expired` on Staging (the user-facing renew/reactivate from 5.1-03 still works; only the automatic flip is dormant). Until step 4, the Vercel Cron route returns 401 in deployed environments (by design — the guard).

## Notes for Downstream Plans

- **Phase 9/10:** the `notifications` table is now actively WRITTEN (`kind='listing_expiring'`). `listMyNotifications`/`unreadNotificationCount` (cookie client, owner RLS) are the durable in-app inbox surface to extend with new `kind`s (chat, reports). New notification kinds need no migration (free-text `kind`); a self-generated notification would need a scoped authenticated INSERT policy (none today — only the service-role cron writes).
- **Notify-once caveat:** v1 keys "already notified" on the existence of ANY `listing_expiring` row for the listing_id. A reactivated/renewed listing whose previous cycle left a `listing_expiring` row will NOT be re-notified on its next near-expiry window. If per-cycle reminders are wanted later, scope the anti-join by `created_at >` the current `expires_at - interval '90 days'` (or store the notified `expires_at`).
