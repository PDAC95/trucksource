## User Constraints (from CONTEXT.md)

### Locked Decisions

**Seller type (ACCT-07)**
- Editable field on the profile/settings page, **optional**. Empty by default = no badge. Zero friction at registration (NOT in the Phase-2 verification onboarding flow).
- **Single-select** — one seller_type per seller. Model as an enum on `profiles_public`, not an array.
- Badge display surfaces: seller's **public profile** (header, near public name) AND **listing card in feed/search**. **NOT on the listing detail page.** The feed/search surface is Phase 7 — in 5.1 prepare the data + a reusable badge component; Phase 7 consumes it.

**Public display name (ACCT-08)**
- A **single free-text "public name" field** — real name OR business name, seller's choice. One public concept that replaces the handle.
- **Reveal flow:** saving a public name triggers an **explicit confirmation modal** ("Your name will be publicly visible… Continue?") with confirm/cancel.
- **Revert:** **reversible** — the original auto-generated anonymous handle is always preserved; revealing only "covers" it. Returning to anonymous brings back the SAME original handle (no new handle generated).
- **Validation:** v1 = basic limits only (max ~50 chars, trim, non-empty). NO blocklist / proactive moderation in v1.
- **Reveal feedback:** on success, toast + preview of the resulting public name.

**90-day expiry & renewal (LIST-09)**
- `expires_at` = publish date + 90 days. Renew OR reactivate = +90 days from now. **Editing a listing does NOT reset the clock.**
- **Sold listings excluded** from the expiry cycle. `sold` is terminal; expiry applies only to `active` listings.
- **Notification:** email + in-app near expiry. Single reminder at **~7 days** before expiry.
- **Renew action surfaces:** inline "Renew" in My Listings AND on the listing detail page (owner-only, when near-expiry/expired). The email notifies but the action happens inside the app (auth-safe) — NOT a one-click renew link in the email.
- **Expired listing — seller view:** appears in My Listings marked "Expired" (dimmed/badge) with a one-click "Reactivate". Buyer never sees it.
- **Reactivation:** direct, no re-validation. One click flips `active` +90 days. Does NOT re-check against current schema.

**Same-seller duplicate warning (LIST-10)**
- Checked **on publish attempt** (server-side, against the caller's own listings).
- Match = same `seller_id` + fuzzy-similar title via `pg_trgm`.
- Warning UI: non-accusatory message + link(s) to the existing similar listing(s).
- **"Publish anyway"** (one extra click) + cancel/edit. NEVER blocks.

**Visible states & feedback**
- Renew/reactivate feedback: toast + new expiry date shown.
- Expiry date visible only when near expiry (~7 days) ("Expires in X days" + Renew). Normal active listings show no countdown.
- My Listings attention indicator: discreet badge/counter ("2 expiring").

## Summary

This phase is a **schema-and-Server-Action evolution of two already-shipped tables** — `profiles_public` (Phase 1, extended by 0002/0009) and `listings` (Phase 5, migration 0006). Nothing here is greenfield. The decisive research finding is that **every pattern this phase needs already exists in the repo and should be copied verbatim, not reinvented**:

- `seller_type` and `display_name` are added to `profiles_public` exactly like `contact_preference` was in `0009_contact_preference.sql` — a `text` column with a CHECK constraint, no new RLS policy (the 0001 owner-update policy already covers owner writes; anon has no update policy). The write path is the `updateContactPreference` Server Action in `lib/actions/account.ts`, cloned.
- `expired` is added to the `listings.status` CHECK exactly as 0006's comment **already anticipated** ("Phase 8 mark-as-sold edits the CHECK… ALTER TYPE ADD VALUE is non-transactional and awkward" — status is deliberately `text + CHECK`, not a pg enum, so adding `expired` is a one-line CHECK edit with **no breaking migration**).
- Duplicate detection reuses the `pg_trgm` extension already enabled in `0001`. Critically, **no trigram index on `listings.title` exists yet** (search is Phase 7 and hasn't shipped any index) — so this phase must add the GIN trigram index itself, and CLAUDE.md invariant #7 forbids `LIKE '%x%'`.

**Primary recommendation:** Ship **one migration** (`0010_stakeholder_lifecycle.sql`) that: adds `seller_type` + `display_name` to `profiles_public`; adds `expires_at timestamptz` to `listings`, extends its status CHECK to include `expired`, backfills `expires_at = date_listed + interval '90 days'` for existing active rows, and adds a partial index `(status, expires_at) where status='active'` for the cron scan plus a `gin (title gin_trgm_ops)` index for duplicate detection; creates a minimal owner-read `notifications` table; and (recommended) installs a **pg_cron daily job** that flips expired `active` rows and a **Vercel Cron → route handler** that sends the ~7-day email + writes the in-app notification. The display-name reveal/revert is a `nullable display_name` that overrides the immutable `username` — the handle column is never mutated, so revert is "set `display_name` back to NULL," which structurally returns the original handle.

|---------|---------|-------------|

| Instead of | Could Use | Tradeoff |

### Pattern 1: Additive column on `profiles_public`, no new RLS policy (ACCT-07, ACCT-08)
**What:** Add `seller_type` and `display_name` as nullable columns on `profiles_public`. Do **not** add an RLS policy — the 0001 `"owner updates own public profile"` policy (`(select auth.uid()) = id`) already authorizes owner writes, and anon has no update policy.
**When to use:** Any new non-PII public-profile field. This is exactly what 0009 did for `contact_preference`.
**Why it's correct here:** `seller_type` names a category and `display_name` is a name the seller *chooses to make public* — neither is PII, neither is copied from `profiles_private`. The cross-cutting privacy gate holds because nothing reads `profiles_private`.
```sql
-- Source: this repo, supabase/migrations/0009_contact_preference.sql (verbatim pattern)
alter table public.profiles_public
  add column if not exists seller_type text
    check (seller_type in (
      'dealer','truck_dismantler','manufacturer','owner_operator',
      'fleet_mechanic','repair_shop','fleet_owner'
    ));   -- nullable, no default → empty = no badge (CONTEXT lock)

alter table public.profiles_public
  add column if not exists display_name text
    check (display_name is null or char_length(btrim(display_name)) between 1 and 50);
```

### Pattern 2: Reveal/revert public name without exposing PII (ACCT-08)
**What:** `username` (the immutable anonymous handle from 0001) is **never mutated** by this feature. `display_name` is a nullable override. The seller's *public name* = `coalesce(display_name, username)`.
**When to use:** The reveal flow sets `display_name`; the revert sets `display_name = NULL`, which structurally returns the **same original handle** (no new handle generated — CONTEXT requirement satisfied by construction).
**Why it's correct:** The handle is preserved because it lives in a different, untouched column. Revert is not "regenerate a handle," it's "drop the override." `display_name` is owner-written through the existing owner-update policy; it is never auto-populated from `profiles_private.first_name`/`last_name`.
```sql
-- public name resolution used by every public read (profile header, listing detail seller)
select coalesce(display_name, username) as public_name, username, seller_type
from public.profiles_public where id = $1;
```
> The 0001 `guard_username_rename` trigger fires only when `username` changes; this feature never changes `username`, so the 30-day rename guard is untouched.

### Pattern 3: Expiry lifecycle on `listings` (LIST-09)
**What:** Add `expires_at timestamptz`; extend the status CHECK from `('active','sold')` to `('active','sold','expired')`; backfill existing active rows to `date_listed + interval '90 days'`. The daily cron flips `active` rows whose `expires_at <= now()` to `expired`. **`sold` is never touched** (terminal). Renew/reactivate set `expires_at = now() + interval '90 days'` and (for reactivate) `status = 'active'`. Editing a listing does NOT touch `expires_at`.
**When to use:** All listing lifecycle transitions.
**Why it's correct:** 0006's comment already planned the CHECK edit. The partial index keeps the cron scan cheap. Public/search reads must add `status = 'active'` so `expired` (and `sold`) never surface to buyers.
```sql
-- Source: extends this repo's 0006_listings.sql status CHECK
alter table public.listings drop constraint listings_status_check;  -- confirm real name first (see Open Q1)
alter table public.listings add constraint listings_status_check
  check (status in ('active','sold','expired'));

alter table public.listings add column if not exists expires_at timestamptz;
update public.listings
  set expires_at = date_listed + interval '90 days'
  where status = 'active' and expires_at is null;

-- cheap daily cron scan: only active rows, ordered by expiry
create index if not exists listings_active_expires_idx
  on public.listings (expires_at) where status = 'active';
```

### Pattern 4: Same-seller fuzzy duplicate detection (LIST-10)
**What:** On publish, before/right after the insert path runs, probe the caller's **own** non-sold listings for a title with high trigram similarity. Show a non-blocking warning with links; never block.
**When to use:** Inside (or just before) `createListing`, scoped to `seller_id = caller`.
**Why it's correct:** Reuses `pg_trgm` (CLAUDE.md #7). Requires a trigram index on `title` (does not exist yet — Phase 7 search hasn't shipped). Threshold tuned for precision (see Open Q2).
```sql
-- Source: PostgreSQL pg_trgm docs (similarity); index required for the % operator / similarity scan
create index if not exists listings_title_trgm_idx
  on public.listings using gin (title gin_trgm_ops);
```

### Pattern 5: Badge data contract for Phase 7 reuse (ACCT-07)
**What:** A single `lib/seller/badge.ts` exporting `SELLER_TYPES` (the 7 values, mirrored from the CHECK) and a `SELLER_TYPE_LABELS` map, consumed by a `<SellerTypeBadge sellerType={...} />` component. Phase 7's feed/search card imports the SAME component + contract.
**When to use:** Anywhere a seller type renders. Mirrors `CONTACT_PREFERENCES` in `lib/account/schema.ts` (single source of truth shared client+server).

### Anti-Patterns to Avoid
- **`LIKE '%title%'` / `ILIKE` for duplicate match.** Violates CLAUDE.md #7, forces a seq scan. Use `similarity()` / `%` with the trigram GIN index.
- **Adding `display_name` to `profiles_private` or reading PII to prefill it.** Breaks the cross-cutting gate. `display_name` is owner-typed, lives on `profiles_public`.
- **`ALTER TYPE … ADD VALUE` (pg enum) for `expired`.** `status` is deliberately `text + CHECK`; just edit the CHECK.
- **Mutating `username` on reveal.** Revert must restore the *same* handle; only `display_name` may change.
- **Letting the duplicate probe block publish.** CONTEXT: never blocks; "Publish anyway" is one extra click.
- **A new RLS policy for the new profile columns.** The 0001 owner-update policy already covers them (0009 precedent).
- **Forgetting `status='active'` on public/search reads.** `expired`/`sold` must be invisible to buyers (ties to Phase 7).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Daily expiry flip scheduling | A Node setInterval / external worker | `pg_cron` `cron.schedule` (in-DB) | Native Supabase; runs regardless of app uptime; pure SQL |
| Fuzzy title matching | Custom Levenshtein in JS, or `LIKE '%x%'` | `pg_trgm` `similarity()` + GIN index | Already enabled; index-backed; CLAUDE.md #7 |
| Owner-scoped write authz | Manual ownership `if` checks in app code | The cookie client + existing owner RLS policies | RLS is the authz boundary (the whole repo's posture) |
| Status enum extension | A new pg enum type / migration dance | Edit the existing `text` CHECK | 0006 chose `text + CHECK` precisely for this |
| Near-expiry email | Hand-rolled SMTP | Resend (already wired) via a Vercel Cron route handler | Existing email path |

**Key insight:** This phase is almost entirely *composition of patterns the repo already proved*. The only genuinely new infrastructure is (a) the pg_cron job, (b) the `notifications` table, and (c) the trigram index on `listings.title`. Everything else is "clone the 0009 + `updateContactPreference` + `getMyListings` patterns."

## Common Pitfalls

### Pitfall 1: `display_name` accidentally sourced from or co-located with PII
**What goes wrong:** A well-meaning "prefill the public name with the seller's real name" reads `profiles_private.first_name`/`last_name` — instantly violating the core privacy guarantee and the phase's cross-cutting gate.
**Why it happens:** "Display name" sounds like "the user's name," which lives in `profiles_private`.
**How to avoid:** `display_name` is **owner-typed free text on `profiles_public`**, empty by default. No code path reads `profiles_private` to populate it. The integration test asserts the PII denylist (`PII_KEYS` in `tests/integration/_supabase.ts`) is still absent from public reads after the column is added.
**Warning signs:** any `select` touching `first_name`/`last_name` in the account display-name flow.

### Pitfall 2: Revert generates a NEW handle instead of restoring the original
**What goes wrong:** Treating reveal as "rename `username` to the real name" means revert can't recover the original handle.
**Why it happens:** Conflating "public name" with the `username` column.
**How to avoid:** Never write `username`. Reveal = set `display_name`; revert = `display_name = NULL`. The handle is structurally preserved. A test reveals then reverts and asserts the resolved public name equals the original `username`.

### Pitfall 3: The cron flips `sold` listings, or flips on the wrong column
**What goes wrong:** A too-broad `update … set status='expired' where expires_at <= now()` catches `sold` rows (which have no clock) or rows already `expired`.
**Why it happens:** Forgetting the `status='active'` guard.
**How to avoid:** `update listings set status='expired' where status='active' and expires_at <= now()`. The partial index encodes the same predicate. A test seeds one active-past-expiry, one sold-past-expiry, one active-future and asserts only the first flips.

### Pitfall 4: Duplicate query degenerates to a sequential scan / false positives
**What goes wrong:** No trigram index → seq scan (slow, and CLAUDE.md #7 violation). Or threshold too low → every "Hood" warns against every other "Hood," training sellers to ignore the warning.
**Why it happens:** Search (Phase 7) hasn't shipped any index on `listings`, so devs assume one exists. Default `pg_trgm` similarity threshold (0.3) is loose.
**How to avoid:** Add `gin (title gin_trgm_ops)`. Confirm with `EXPLAIN ANALYZE`. Tune the threshold for **precision** (start ~0.6, validate against the real launch dataset; see Open Q2). Scope strictly to `seller_id = caller and status <> 'sold'`.

### Pitfall 5: `expired` listings leak into public surfaces
**What goes wrong:** `getListing`/feed/search return `expired` rows because the read doesn't filter status. Buyers see dead inventory; the "hidden from search" requirement fails.
**Why it happens:** 0006 made `listings` public-read on ALL rows (RLS `using (true)`); status filtering is the **app's** job, not RLS's.
**How to avoid:** Every buyer-facing read filters `status = 'active'`. `getMyListings` (owner) keeps all statuses. Phase 7 search must inherit this — note it explicitly in the handoff. A test fetches an `expired` listing as anon via the public read path and asserts it's excluded.

### Pitfall 6: Vercel Cron route handler is unauthenticated
**What goes wrong:** The `/api/cron/near-expiry` endpoint is publicly callable → anyone can trigger mass emails (spam / Resend bill).
**Why it happens:** Route handlers are public by default.
**How to avoid:** Guard with the Vercel Cron secret (`Authorization: Bearer ${CRON_SECRET}` header check) and/or `vercel.json` cron config; reject otherwise. Server-only; never `NEXT_PUBLIC_`. (Same spirit as the Phase-2 OTP endpoint guards.)

### 2. updateDisplayName Server Action (clone of updateContactPreference)
```typescript
// Source: this repo, lib/actions/account.ts (updateContactPreference) — cloned shape
"use server";
import { createClient } from "@/lib/supabase/server";
import { displayNameSchema } from "@/lib/account/schema";

export async function updateDisplayName(input: unknown) {
  const supabase = await createClient();
  const { data: claims } = await supabase.auth.getClaims();
  const userId = claims?.claims?.sub;          // getClaims, never getSession (invariant #6)
  if (!userId) return { ok: false, error: "unauthenticated" } as const;

  const parsed = displayNameSchema.safeParse(input);
  if (!parsed.success) return { ok: false, error: "invalid" } as const;
  // displayName === null → revert to anonymous handle (no username write).
  const { data, error } = await supabase
    .from("profiles_public")
    .update({ display_name: parsed.data.displayName ?? null })
    .eq("id", userId)
    .select("id, username, display_name");      // return resolved name for the preview toast
  if (error) return { ok: false, error: "invalid" } as const;
  if (!data || data.length === 0) return { ok: false, error: "not_found" } as const;
  return { ok: true, publicName: data[0].display_name ?? data[0].username } as const;
}
```

### 7. renewListing / reactivateListing Server Actions (owner-scoped, zero-rows → not_found)
```typescript
// Source: updateListing in lib/actions/listings.ts (owner-scoped, zero-rows = not_found)
export async function renewListing(id: number) {
  const supabase = await createClient();
  const { data: claims } = await supabase.auth.getClaims();
  const userId = claims?.claims?.sub;
  if (!userId) return { ok: false, error: "unauthenticated" } as const;
  // renew only an ACTIVE listing; +90 days from now. Editing never calls this.
  const { data, error } = await supabase
    .from("listings")
    .update({ expires_at: new Date(Date.now() + 90*864e5).toISOString() })
    .eq("id", id).eq("seller_id", userId).eq("status", "active")
    .select("id, expires_at");
  if (error) return { ok: false, error: "invalid" } as const;
  if (!data?.length) return { ok: false, error: "not_found" } as const;
  return { ok: true, expiresAt: data[0].expires_at } as const;
}
// reactivateListing: same, but .eq("status","expired") and set status:'active' + expires_at.
```

Accompanying SQL function (SECURITY INVOKER so RLS still applies, scoped to caller):
```sql
create or replace function public.find_similar_own_listings(p_title text, p_threshold real)
returns table (id bigint, title text, sim real)
language sql stable security invoker set search_path = '' as $$
  select l.id, l.title, similarity(l.title, p_title) as sim
  from public.listings l
  where l.seller_id = (select auth.uid())
    and l.status <> 'sold'
    and similarity(l.title, p_title) >= p_threshold
  order by sim desc limit 5;
$$;
```

## In-App Notification Surface (Open Question resolved)

**Does a notifications table exist?** No. Grep of `supabase/migrations/*` shows tables for profiles, verification/abuse, fitment, garage, listings (+ fitment/photos/view_events) — **no notifications table**. CONTEXT locks **email + in-app**, so this phase must create a minimal one (schema above). It is owner-read RLS, written by the near-expiry Vercel Cron route handler (service-role, since the job acts on behalf of the system, not a logged-in user). `kind` is a free `text` so Phase 9 (chat) and Phase 10 (admin) can add notification kinds non-breakingly. My Listings reads it (unread count → the "2 expiring" badge can also be derived directly from `expires_at`, so the in-app notification row is the durable record + a future inbox surface).

## Open Questions

1. **Exact name of the `listings` status CHECK constraint.**
   - What we know: 0006 created `status text not null default 'active' check (status in ('active','sold'))` as an inline column CHECK. Inline CHECKs get an auto-generated name (commonly `listings_status_check`).
   - What's unclear: whether Staging's actual constraint name matches (inline checks are sometimes named `listings_status_check1` etc.).
   - Recommendation: in the migration, query `\d public.listings` (or `pg_constraint`) to confirm the name, or use a DO block that drops by discovered name. Do NOT hardcode blindly. Apply non-destructively via `supabase db query --linked -f` (the repo's established apply path).

2. **pg_trgm similarity threshold for the duplicate warning.**
   - What we know: default `pg_trgm.similarity_threshold` is 0.3 (loose); CONTEXT wants precision (avoid false positives).
   - What's unclear: the right value for short part titles ("Hood Peterbilt 379" vs "Peterbilt 379 Hood"). Reordered-word variants score lower on raw trigram similarity than substring variants.
   - Recommendation: start at **~0.6**, validate against the real launch dataset during planning/UAT, and make it a function parameter (as in Example 9) so it's tunable without a migration. Consider `word_similarity()` if word-order robustness matters more than raw similarity.

3. **Should the daily flip be pg_cron (DB) or Vercel Cron (app)?**
   - What we know: the flip is pure SQL; the notify needs Resend + an in-app insert.
   - Recommendation: **split** — pg_cron for the SQL flip (Example 4), Vercel Cron → route handler for the ~7-day notify (email + in-app row). If the team prefers one mechanism, Vercel Cron can do both (the route handler runs the flip UPDATE too), but then the flip depends on app uptime. Decide at planning; both are non-breaking.

4. **Where to invoke the duplicate probe relative to `createListing`.**
   - What we know: CONTEXT says "checked on publish attempt… warning shown before the listing is confirmed."
   - Recommendation: run `find_similar_own_listings` as a separate Server Action the form calls on submit; if matches return, show the non-blocking dialog; "Publish anyway" then calls the existing `createListing`. This keeps `createListing` unchanged and the probe strictly advisory.
