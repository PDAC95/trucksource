<domain>
## Phase Boundary

Fold in four stakeholder trust/lifecycle features (check.md, 2026-06-08) on top of already-verified phases, WITHOUT changing the privacy model or seller permissions:

1. **Seller type badge** (ACCT-07) — informational-only label (Dealer, Truck Dismantler, Manufacturer, Owner Operator, Fleet Mechanic, Repair Shop, Fleet Owner). Changes nothing about permissions.
2. **Opt-in public display name** (ACCT-08) — anonymous auto-handle by default; seller may deliberately reveal a real/business name that *replaces* the handle.
3. **90-day frictionless expiry/renewal** (LIST-09) — listings auto-expire 90 days after publish/renewal; near-expiry notification; one-click renew; unrenewed → `expired` (hidden from search, NOT deleted, one-click reactivable).
4. **Soft same-seller duplicate warning** (LIST-10) — non-blocking warning when the same seller posts a similar listing; always allowed to publish.

Privacy invariant (re-verified gate): `seller_type` and the chosen public display name live on `profiles_public` (non-PII, owner-written); the public name is NEVER auto-populated from `profiles_private`. `expired` status and `expires_at` add no public PII.

Out of scope: ratings & reviews (REP-01/02 stay v2); LIST-08 min-3-photos (handled as a separate Phase 5 gap); the flat part-category catalog load (data task, not this phase).

<decisions>
## Implementation Decisions

### Seller type (ACCT-07)
- **Placement:** editable field on the profile/settings page, **optional**. Empty by default = no badge. Zero friction at registration; seller adds it when they want. (Not in the verification onboarding flow — keeps Phase 2's flow untouched.)
- **Cardinality:** **single-select** — one seller_type per seller. Model as an enum on `profiles_public`, not an array. One clear badge for the buyer.
- **Badge display surfaces:** seller's **public profile** (header, near public name) AND **listing card in feed/search**. NOTE: NOT on the listing detail page. The feed/search surface is Phase 7 — in 5.1 prepare the data + a reusable badge component; Phase 7 consumes it for the feed render.

### Public display name (ACCT-08)
- **Field shape:** a **single free-text "public name" field** — the seller types either their real name OR a business name, their choice. One public concept (consistent with the stakeholder decision: single public name replaces the handle).
- **Reveal flow:** saving a public name triggers an **explicit confirmation modal** — "Your name will be publicly visible on all your listings and profile. Continue?" with confirm/cancel. Deliberate action, hard to reveal by accident.
- **Revert:** **yes, reversible** — the original auto-generated anonymous handle is always preserved; revealing only "covers" it. Seller can return to anonymous anytime and the SAME original handle comes back (no new handle generated).
- **Validation:** v1 = basic limits only (max ~50 chars, trim, non-empty). NO blocklist / proactive moderation in v1 — abuse handled later via report/admin (Phase 9/10). Consistent with the "no AI moderation in v1" stance.
- **Reveal feedback:** on success, toast + **preview of the resulting public name** ("Your public name is now: X") so the seller sees exactly what buyers see.

### 90-day expiry & renewal (LIST-09)
- **Clock:** `expires_at` = publish date + 90 days. Renew OR reactivate = +90 days from now. **Editing a listing does NOT reset the clock** (prevents trivial-edit immortality).
- **Sold listings:** **excluded** from the expiry cycle. `sold` is a terminal v1 state (off-platform sale); no clock, no expiry notification. Expiry applies only to `active` listings.
- **Notification:** **email + in-app** near expiry (Resend SMTP already wired). Single reminder at **~7 days** before expiry (enough margin for one-click renew, no spam).
- **Renew action surfaces:** inline **"Renew" button in My Listings** AND a **"Renew" button on the listing detail page** (visible to the owner when near-expiry or expired). The email *notifies* but the action happens inside the app (auth-safe) — NOT a direct one-click renew link in the email.
- **Expired listing — seller view:** appears in My Listings marked **"Expired"** (dimmed/badge) with a one-click **"Reactivate"** button. Buyer never sees it (hidden from search), nothing is deleted.
- **Reactivation:** **direct, no re-validation** — data + photos are intact, so one click flips it back to `active` +90 days. Maximum no-friction (the governing principle of this feature). Does NOT re-check against current schema (e.g. min-3-photos) even if rules changed.

### Same-seller duplicate warning (LIST-10)
- **Trigger:** checked **on publish attempt** (server-side, against the caller's own listings). Warning shown before the listing is confirmed. Single query, no live-typing noise.
- **Match definition:** same `seller_id` + **fuzzy-similar title** via `pg_trgm` (already installed). Catches reordered/typo'd variants (e.g. "Hood Peterbilt 379" vs "Peterbilt 379 Hood"). Consistent with the search stack.
- **Warning UI:** non-accusatory message + **link(s) to the existing similar listing(s)** so the seller can view/edit the original. Tone: "If it's the same one, edit the original; if it's a different part, go ahead."
- **Proceed:** the warning offers **"Publish anyway"** (one extra click) + cancel/edit. NEVER blocks — but the extra click confirms intent. (A seller legitimately may have multiple similar parts.)

### Visible states & feedback
- **Renew/reactivate feedback:** toast + the **new expiry date shown** ("Renewed — active until DD/MM"); the listing reflects the new date.
- **Expiry date visibility:** shown **only when near expiry (~7 days)** ("Expires in X days" + Renew). Normal active listings show no countdown — don't remind the seller of something the system handles automatically.
- **My Listings attention indicator:** a discreet **badge/counter** ("2 expiring") when listings need attention (expiring/expired). Reuses the data already computed; helps without overwhelming.

### Claude's Discretion
- Exact enum values vs labels mapping for seller_type (the 7 types are fixed; display formatting is open).
- Visual design of the badge component, the confirmation modal, toasts, and the dimmed "Expired" treatment.
- pg_trgm similarity threshold tuning for the duplicate match (must be tuned for precision — avoid false-positive warnings).
- Exact copy/wording of notifications, modal, and warning (intent is captured above).
- Scheduling mechanism details for the expiry flip (stakeholder memory suggests Supabase pg_cron — confirm during planning).
- Whether the near-expiry in-app indicator is a banner, badge, or row state.
</decisions>

<specifics>
## Specific Ideas

- The whole phase is governed by a **no-friction-for-the-seller** principle (explicit stakeholder constraint on expiry, extended here to renew/reactivate/duplicate flows). When a decision was close, the lower-friction option won.
- Duplicate-warning tone must **remind, never accuse** — the seller may legitimately have several identical parts.
- The reveal-name modal must make revealing identity feel like a **deliberate, reversible** privacy choice, not a trap.
- Builds on existing infra: `profiles_public` table (Phase 1), `is_verified_seller` boolean (Phase 2), listings `status` text+CHECK enum built extensible in Phase 5 (only the CHECK is edited to add `expired` — no breaking migration), `pg_trgm` + Resend SMTP already in place.
</specifics>

<deferred>
## Deferred Ideas

- **Multiple seller types per seller** — considered, rejected for v1 (single-select chosen). Could revisit if stakeholders want richer trust signals.
- **Blocklist / moderation on public names** — deferred to the report/admin path (Phase 9/10); v1 ships basic length/trim limits only.
- **Live duplicate detection while typing the title** — deferred; v1 checks on publish only.
- **Direct one-click renew link in the notification email** — deferred (auth-flow surface); v1 keeps the renew action inside the app.
- **Badge render in the feed/search results** — the *data + component* are prepared here, but the actual feed render is owned by **Phase 7** (Search, Feed & Public Profile).
- **Sold listings expiring** — rejected for v1 (sold stays terminal, out of the clock).
- **Image/AI-similarity duplicate detection** — out of scope (v1 "no AI moderation").
- **Ratings & reviews (REP-01/02)** — stays v2 per stakeholder confirmation.
</deferred>
