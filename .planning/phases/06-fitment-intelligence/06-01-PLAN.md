**1) public.fitment_rules** — the rules backbone, modelled EXACTLY on `search_term_targets` (exclusive-arc of real FKs, NOT a `trigger_value text` discriminator — RESEARCH State-of-the-Art + Anti-Patterns):
  - TRIGGER arms (nullable FKs, exactly one set): `trigger_category_id bigint references public.part_categories(id) on delete cascade`, `trigger_model_id bigint references public.models(id) on delete cascade`, `trigger_config_id bigint references public.configurations(id) on delete cascade`.
  - IMPLIES arms (nullable FKs, exactly one set): `implies_model_id -> public.models`, `implies_config_id -> public.configurations`, `implies_category_id -> public.part_categories`, `implies_search_term_id -> public.search_terms`, `implies_special_filter_id -> public.special_filters` (all `on delete cascade`).
  - `confidence smallint not null default 100`, `created_at timestamptz not null default now()`.
  - CHECKs: `constraint exactly_one_trigger check (num_nonnulls(trigger_category_id, trigger_model_id, trigger_config_id) = 1)`; `constraint exactly_one_implies check (num_nonnulls(implies_model_id, implies_config_id, implies_category_id, implies_search_term_id, implies_special_filter_id) = 1)`; `constraint confidence_range check (confidence between 0 and 100)`.
  - Indexes: `fitment_rules_trigger_category_idx`, `fitment_rules_trigger_model_idx`, `fitment_rules_trigger_config_idx`.
  - **Idempotent-seed enabler:** add a unique index folding all NULL arms so 0013 can `on conflict do nothing` (the repo's coalesce(...,0) trick): `create unique index fitment_rules_uniq on public.fitment_rules (coalesce(trigger_category_id,0), coalesce(trigger_model_id,0), coalesce(trigger_config_id,0), coalesce(implies_model_id,0), coalesce(implies_config_id,0), coalesce(implies_category_id,0), coalesce(implies_search_term_id,0), coalesce(implies_special_filter_id,0));`
  - RLS: `alter table public.fitment_rules enable row level security;` then ONE policy `"fitment_rules public-readable" ... for select to anon, authenticated using (true);`. **NO insert/update/delete policy** → writes are seed/service-role only (exactly like every Phase-3 reference table). Do NOT add a `using (true)` write policy (Pitfall 3 warning sign).
