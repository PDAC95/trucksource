  <name>Task 1: Migration 0014_search.sql — tsvector + GIN + search_events + search_listings RPC</name>
  <files>supabase/migrations/0014_search.sql</files>
  <action>
Create `supabase/migrations/0014_search.sql` following the 0012 header-comment + RLS-in-migration conventions (lowercase SQL, RLS enabled in THIS migration per CLAUDE.md invariant #2, `search_path=''` on functions per repo convention).

1. **tsvector generated column over the LOCKED FTS source set.** Per stakeholder decision the FTS text source is: part title, part number, the Phase-6 Common Search Terms / slang tags, and `damage_notes`. Do NOT add a `description` column; do NOT touch the listing form. Because slang tags live in a child table (`listing_search_terms` -> `search_terms.term`), a generated column cannot aggregate them — so build the generated column over the row-local immutable fields and fold slang into the RPC:
   ```sql
   alter table public.listings
     add column if not exists search_vector tsvector
     generated always as (
       to_tsvector('english',
         coalesce(title,'') || ' ' ||
         coalesce(part_number,'') || ' ' ||
         coalesce(damage_notes,''))
     ) stored;
   ```
   Use plain `to_tsvector('english', …)` — it IS immutable. Do NOT wrap in `unaccent()` (Pitfall 2: unaccent is STABLE, not IMMUTABLE — a generated column using it fails to create). pg_trgm covers fuzzy/accent fallback.

2. **GIN indexes:**
   ```sql
   create index if not exists listings_search_vector_idx on public.listings using gin (search_vector);
   ```
   A trigram GIN index on `listings.title` already exists (0010 `listings_title_trgm_idx`) — reuse it for fuzzy title match; do NOT recreate. Add `create index if not exists search_terms_term_trgm_idx on public.search_terms using gin (term gin_trgm_ops);` (cast citext if needed) so the slang typo-match via `public.similarity(term::text, p_q)` is index-backed (this index also backs the bare `%` operator, but the RPC/readers use `public.similarity()` for `search_path=''` safety).

3. **search_events table — clone listing_view_events' insert-only RLS posture EXACTLY:**
   ```sql
   create table public.search_events (
     id bigint generated always as identity primary key,
     raw_term text,
     normalized_term text,
     facets jsonb not null default '{}'::jsonb,
     result_count int not null,
     searcher_id uuid references auth.users(id) on delete set null,  -- NULL = anon
     created_at timestamptz not null default now()
   );
   alter table public.search_events enable row level security;
   create policy "search_events insert" on public.search_events
     for insert to anon, authenticated with check (true);
   ```
   EXACTLY ONE policy (insert). NO select/update/delete → service-role-only read (Phase 10). Captures raw + normalized term, applied facets (jsonb), result count, nullable searcher (SRCH-05 capture list).

4. **search_listings RPC** — `language sql stable security invoker set search_path = ''`, returns ONLY public columns (NEVER seller PII — Pitfall 1). Signature and body per 07-RESEARCH Pattern 1, with these facet args: `p_q text`, `p_model_id`, `p_config_id`, `p_category_id`, `p_condition_id`, `p_fits_model_id`, `p_fits_config_id`, `p_limit int default 24`, `p_offset int default 0`. Do NOT add `p_material_id` / `p_special_filter_id` args (those facets are DEFERRED — see decision note below). Body rules:
   - `where l.status = 'active'` AND non-expired: `(l.expires_at is null or l.expires_at > now())` (excludes drafts/expired/sold per CONTEXT).
   - Keyword arm matches the tsvector OR the per-listing slang tags so a listing tagged with slang matches even when the title doesn't contain it:
     ```sql
     and (p_q is null or
          l.search_vector @@ websearch_to_tsquery('english', p_q)
          or exists (select 1 from public.listing_search_terms lst
                     join public.search_terms st on st.id = lst.term_id
                     where lst.listing_id = l.id
                       and public.similarity(st.term::text, p_q) >= 0.3))
     ```
     IMPORTANT — use the schema-qualified FUNCTION form `public.similarity(a, b) >= <threshold>`, NOT the bare `%` operator. Under `set search_path = ''` the bare `%` trigram operator may fail to resolve (pg_trgm is installed in the `public` schema). Migration 0010's `find_similar_own_listings` (lines ~112–122) sets the precedent — mirror it exactly. Cast the citext term to text (`st.term::text`). The 0.3 threshold matches pg_trgm's default `similarity_threshold`; the `search_terms_term_trgm_idx` GIN index still backs `similarity()` on the typo path. Do NOT use `st.term % p_q`.
   - condition arm: `(p_condition_id is null or l.condition_id = p_condition_id)`.
   - model/config arm via EXISTS on `listing_fitment` (config null = match any config of that model).
   - category arm via EXISTS on `listing_categories`.
   - fits-my-truck arm via EXISTS on `listing_fitment` with `(p_fits_config_id is null or lf.config_id is null or lf.config_id = p_fits_config_id)`.
   - `rank` column: `case when p_q is null then 0 else ts_rank(l.search_vector, websearch_to_tsquery('english', p_q)) end`.
   - `order by rank desc, l.date_listed desc` (relevance first, recency tiebreaker — CONTEXT).
   - `limit p_limit offset p_offset`.
   Return columns: `id bigint, title text, asking_price numeric, condition_id bigint, date_listed timestamptz, rank real, total_count bigint` (the reader resolves photos/fitment/seller separately — keep the RPC PII-free and column-minimal).
   - **Total result count (LOCKED CONTEXT "X resultados") — single-query, deterministic:** add a window `count(*) over()` column aliased `total_count` to the SELECT. Because `count(*) over()` is computed over the full filtered set BEFORE `limit/offset`, every returned row carries the identical grand total — so the 07-02 reader reads `total` from `rows[0].total_count` (and treats no-rows as `0`). This is the ONE chosen strategy: no second round-trip, no `p_limit:1000` re-call, no `cards.length` approximation.

5. **`explain_search_plan` helper — NON-optional, created in THIS migration (cross-cutting EXPLAIN gate path).** The EXPLAIN ANALYZE index check (Pitfall 3) is a cross-cutting gate and must have a guaranteed, reliably-runnable path through PostgREST — not an "if a raw EXPLAIN can't run" afterthought. Always create:
   ```sql
   create or replace function public.explain_search_plan(p_q text)
   returns setof text
   language plpgsql stable security invoker set search_path = '' as $$
   begin
     return query execute
       'explain analyze select * from public.listings ' ||
       'where search_vector @@ websearch_to_tsquery(''english'', $1)'
       using p_q;
   end;
   $$;
   ```
   It returns ONLY plan text (no listing rows, no PII). Task 2's index assertions call `rpc('explain_search_plan', { p_q: 'bumper' })` and grep the returned plan text. Keep it security invoker + `search_path=''` per repo convention.

6. **Slang helper RPCs for the lib/search readers — `match_search_term` + `autocomplete_terms` (consumed by 07-02).** So 07-02's `expandSlang`/`autocomplete` never reach for the bare `%` operator (which may not resolve under `search_path=''`), expose two trgm-index-backed readers here:
   ```sql
   create or replace function public.match_search_term(p_raw text)
   returns table (id bigint, term text)
   language sql stable security invoker set search_path = '' as $FN$
     select st.id, st.term::text
     from public.search_terms st
     where public.similarity(st.term::text, p_raw) >= 0.3
     order by public.similarity(st.term::text, p_raw) desc
     limit 1;
   $FN$;

   create or replace function public.autocomplete_terms(p_prefix text)
   returns table (term text)
   language sql stable security invoker set search_path = '' as $FN$
     select st.term::text
     from public.search_terms st
     where public.similarity(st.term::text, p_prefix) >= 0.3
     order by public.similarity(st.term::text, p_prefix) desc
     limit 6;
   $FN$;
   ```
   Both are anon-callable (public reads, no PII), `public.similarity()` not bare `%`, and exercise `search_terms_term_trgm_idx`. (Use the `$FN$` dollar-tag literally as shown.)
