  <action>
1. Install the missing shadcn components via the project CLI: `npx shadcn@latest add dialog alert-dialog card` (these don't exist yet; `select`/`button`/`form`/`input`/`label` already do, radix-ui@1.4.3 backs them). Do NOT add `command`/`popover` — launch list sizes (2 makes / 17 models / 9 configs) are small enough that plain `select` suffices per CONTEXT's "decide per-level by actual list length."

2. Add cascade data fetchers. Since the reference tables are anon-public read, the cleanest repo-aligned approach: fetch the dependent options server-side. Add two small `"use server"` reader functions — put them in `lib/garage/cascade.ts` (a new file owned by THIS task; add it to files if needed) OR inline as server actions:
   - `getModels(makeId: number): Promise<{id:number;name:string}[]>` → `from("models").select("id,name").eq("make_id", makeId).order("name")`.
   - `getConfigs(modelId: number): Promise<{id:number;name:string}[]>` → `from("configurations").select("id, name, model_configurations!inner(model_id)").eq("model_configurations.model_id", modelId).order("name")` (scope configs THROUGH model_configurations applicability — NEVER show all 9 configs; Pitfall 2). Return `{id,name}[]`.
   (Initial makes are loaded by the page in Task 2 and passed as props.)

3. **truck-cascade.tsx** (`"use client"`): the dependent cascade form, RHF + `zodResolver(truckSchema)`, three shadcn `Select`s:
   - Make select (from `makes` prop). On change: load models via `getModels`, reset model+config selections.
   - Model select (from loaded models; disabled until a make is chosen). On change: load configs via `getConfigs`, reset config.
   - Config select (from loaded configs; optional — include an explicit "No specific configuration" option that maps to `configId = null`; disabled until a model is chosen).
   - Nickname `Input` (optional, maxLength 40).
   Expose props: `makes: {id;name}[]`, `defaultValues?` (for edit pre-fill — when editing, pre-select make derived from the truck's makeId, pre-load its models/configs, and set model/config/nickname), and an `onSubmit(values: TruckInput)` callback the parent dialog supplies. Show a loading/disabled state while a level fetches (Claude's discretion on skeleton). The form ONLY produces values; the dialog calls the action. Serialize Radix Select string values — truckSchema's `z.coerce.number()` handles the coercion server-side, but coerce in the form too for the submit payload shape.
  </action>
