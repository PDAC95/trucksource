# Phase 05.1 Plan 01: Stakeholder Lifecycle Foundation Summary

One migration (0010) lands all four stakeholder-trust/lifecycle features' schema plus the shared `lib/seller/badge.ts` badge contract, the extended account Zod schema, and the Wave-0 privacy/correctness test gates — built directly against the shipped `profiles_public` (0001/0009) and `listings` (0006) tables with the cross-cutting PII gate intact.

## What Shipped

**Task 1 — contracts (commit `0f72525`):**
- `lib/seller/badge.ts`: `SELLER_TYPES` (7 fixed values), `SELLER_TYPE_LABELS`, `SellerType`, and `resolvePublicName(displayName, username)` = `coalesce(display_name, username)`. The Phase-7 feed/search card imports this same contract (no duplication).
- `lib/account/schema.ts`: appended `sellerTypeSchema` (`z.enum(SELLER_TYPES).nullable()`) and `displayNameSchema` (`z.string().trim().min(1).max(50).nullable()`) + inferred `SellerTypeInput`/`DisplayNameInput`. Both nullable — clearing the badge / reverting to the anonymous handle is valid.

**Task 2 — migration 0010 (commit `d2327e3`, applied + idempotently re-applied + verified on Staging):**
- `profiles_public`: additive `seller_type` (text+CHECK, nullable, no default) and `display_name` (text, `null or 1-50 trimmed`). NO new RLS policy — the 0001 owner-update policy covers them, anon has none (0009 precedent).
- `listings`: status CHECK extended `('active','sold')` → `('active','sold','expired')`; `expires_at timestamptz` added; existing active rows backfilled to `date_listed + 90 days` (0 active rows left null).
- Indexes: `listings_active_expires_idx` (partial `where status='active'`) for the cron scan; `listings_title_trgm_idx` (gin `gin_trgm_ops`) for duplicate match.
- `notifications` table: owner-read + owner-update RLS, **NO authenticated INSERT policy** (service-role cron writes rows in 5.1-05); `(user_id, created_at desc)` index.
- `find_similar_own_listings(p_title, p_threshold)`: `language sql stable security invoker set search_path = ''`, scoped to caller's non-sold listings.

**Task 3 — Wave-0 gates (commit `5312057`, full suite 23 files / 117 passed / 1 skipped):**
- `_supabase.ts` `PUBLIC_PROFILE_KEYS` extended with `contact_preference`, `seller_type`, `display_name`.
- `stakeholder-lifecycle.test.ts` (integration, ran live vs Staging): PII-absence on the public profile read, `seller_type ∈ SELLER_TYPES|null`, anon-update changes zero rows, `status='active'` read never yields a non-active row.
- `display-name.test.ts`: reveal covers handle, revert (`null`) restores the SAME original `username`.
- `expire-flip.test.ts` + `lib/listings/lifecycle.ts` `shouldExpire`: active-past→true; sold-past / active-future / expired-past→false.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] `find_similar_own_listings` failed to create as written**
- **Found during:** Task 2 (first migration apply)
- **Issue:** The research/plan SQL used an unqualified `similarity(...)`. With `set search_path = ''` Postgres could not resolve it (`ERROR 42883: function similarity(text, text) does not exist`).
- **Fix:** Confirmed pg_trgm is installed in the `public` schema on this project, then schema-qualified both calls to `public.similarity(...)`. Re-applied cleanly and idempotently.
- **Files modified:** supabase/migrations/0010_stakeholder_lifecycle.sql
- **Commit:** d2327e3

Note: the `listings_status_check` constraint name (Open Q1) was confirmed via `pg_constraint` before the drop — it matched the plan's assumed name, so the `drop constraint if exists` was a no-op risk either way.
