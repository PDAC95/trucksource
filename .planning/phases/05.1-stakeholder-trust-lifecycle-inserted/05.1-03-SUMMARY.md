# Phase 05.1 Plan 03: 90-Day Expiry/Renewal Lifecycle Summary

Shipped the user-facing half of LIST-09: owner-scoped `renewListing` (+90d, active-only) and `reactivateListing` (+90d, status→active, expired-only) Server Actions, the `isExpiringSoon`/`daysUntil` pure helpers, lifecycle-aware reads (`getMyListings` derives `expiringSoon`; the public page hides expired/sold from buyers), and a self-hiding `RenewButton` surfaced in My Listings + on the owner's listing detail. Editing still never moves the 90-day clock; the DB-side daily flip and ~7-day notify remain 5.1-05.

## What Shipped

**Task 1 — actions + helpers (commit `dbfe27d`):**
- `lib/listings/lifecycle.ts`: `daysUntil(expiresAt, now?)` (ceil; can be 0/negative) and `isExpiringSoon(status, expiresAt, now?)` = `status==='active' && expiresAt && 0 < d <= 7` — a healthy active listing (>7 days) is NOT "expiring soon" (CONTEXT: no countdown on healthy listings).
- `lib/actions/listings.ts`: `renewListing(id)` — owner-scoped `.update({expires_at: now+90d}).eq(id).eq(seller_id).eq(status,'active').select(...)`; zero rows → `not_found`. `reactivateListing(id)` — same but `.eq(status,'expired')` and `.update({status:'active', expires_at})`. Both return the new `expiresAt`. A header note records that `updateListing` deliberately never touches `expires_at`.

**Task 2 — lifecycle-aware reads (commit `a09635f`):**
- `getMyListings` selects `expires_at`, the `MyListing` type gains `expiresAt` + derived `expiringSoon` (via `isExpiringSoon`); the owner list keeps ALL statuses so expired rows are reactivable.
- `getListing`'s `ListingDetail` gains `expiresAt`; `getListing` stays status-agnostic (documented) so the owner edit/detail path can read its own non-active listing.
- `app/(public)/listings/[id]/page.tsx`: after `getListing`, `notFound()` when `status !== 'active'` — the app-layer status filter (RLS is public-read on all rows; Pitfall 5). The active check runs BEFORE `recordListingView`, so views log only for visible listings.

**Task 3 — UI + attention states + test (commit `f2bbb49`):**
- `components/listings/renew-button.tsx` (`"use client"`): self-hiding control. Renders nothing on healthy active rows; "Expires in X days" + Renew when expiring; Reactivate when expired. Runs the action in a transition, sonner-toasts the new expiry (`Renewed/Reactivated — active until DD/MM/YYYY`), then `router.refresh()`.
- `app/(app)/sell/listings/page.tsx`: per-row `<RenewButton>`, dimmed expired rows + destructive "expired" badge, and a discreet "N expiring" counter near the heading (`expiringSoon || status==='expired'`). Toaster added.
- `components/listings/listing-detail.tsx`: owner-only `<RenewButton>` behind a new `isOwner` prop (buyer view unchanged). The public page computes `isOwner` via getClaims sub === seller_id and mounts a Toaster only for the owner.
- `tests/unit/listing-lifecycle-actions.test.ts` (mocked client, 7 tests): renew unauthenticated short-circuit, renew scoped to `seller_id`+`status='active'` and payload is exactly `{expires_at}`, renew zero-rows→not_found, invalid id; reactivate scoped to `status='expired'` and sets `status:'active'`, zero-rows→not_found; **editing never renews** — `updateListing`'s update payload has no `expires_at` (and no `status`).
