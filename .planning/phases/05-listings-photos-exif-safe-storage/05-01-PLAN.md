Create `lib/images/strip.ts` EXACTLY per 05-RESEARCH Pattern 1:
- `import "server-only";` at the top (keeps sharp server-only — invariant #3).
- `const MAX_BYTES = 10 * 1024 * 1024;` and `const ACCEPTED = ["image/jpeg","image/png","image/webp"];`.
- Export `StripResult` (the discriminated union in the interfaces block) and `stripAndReencode(input: Buffer, declaredType: string)`.
- Order: reject if `input.byteLength > MAX_BYTES` → `too_large`; reject if `!ACCEPTED.includes(declaredType)` → `unsupported_type` (this is how HEIC is rejected — its MIME is not in ACCEPTED).
- Then in a try/catch: `const buffer = await sharp(input).rotate().webp({ quality: 82 }).toBuffer();`
- CRITICAL (invariant #4): the chain has `.rotate()` with NO args (bakes orientation into pixels) and ABSOLUTELY NO `.withMetadata()`, `.keepMetadata()`, or `.keepExif()` — those re-attach EXIF and silently break the gate.
- Also sniff the real format: before re-encoding, call `await sharp(input).metadata()` and if `meta.format` is not one of `jpeg|png|webp`, return `unsupported_type` (don't trust the declared MIME — a .jpg-labeled HEIC must be rejected). On any sharp throw → `decode_failed`.
- Return `{ ok: true, buffer, contentType: "image/webp", ext: "webp" }`.

Create `tests/unit/exif-strip.test.ts` per 05-RESEARCH "No-GPS regression test":
- Build an input image WITH GPS using `sharp({create:{...}}).withExif({ IFD0:{Make:"TestCam"}, GPS:{GPSLatitude:"50/1 17/1 0/1", GPSLatitudeRef:"N"} }).jpeg().toBuffer()`.
- Sanity-assert `await exifr.gps(withGps)` is truthy (the input really has GPS).
- Run `stripAndReencode(withGps, "image/jpeg")`, assert `ok === true`.
- Assert `await exifr.gps(out.buffer)` is FALSY and `(await exifr.parse(out.buffer,{gps:true,exif:true,ifd0:true}))?.latitude` is undefined.
- Second test: `stripAndReencode(Buffer.from([0,1,2]), "image/heic")` returns `{ ok:false, error:"unsupported_type" }`.
This is a `@vitest-environment node` test (sharp needs node). It is the CI gate for LIST-03.
