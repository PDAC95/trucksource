<objective>
Add the per-listing YEAR dimension to the data + search layer. Today year exists only on `garage_trucks`; listings have no year. This plan ships the foundation the two UI plans (16-06 search, 16-07 create/edit) build on: a migration that adds `year_start`/`year_end` to `listings` and recreates `search_listings` with a `p_year` filter arm, a shared year-options helper, the search URL `year` param, and an integration test.

Semantics (stakeholder-confirmed): a listing either fits ALL years (both columns null = universal) or fits a year range `year_start..year_end`; a "specific year" is stored as `year_start = year_end`. The buyer filters by a SINGLE year (their truck's year); a listing matches when `p_year` is null, or the listing is universal, or `p_year` falls inside the listing's range. Existing listings have no year set → they stay universal and remain findable.

Output: migration 0026, `lib/listings/years.ts`, `year` wired through `lib/search/params.ts` + `lib/search/queries.ts`, and `tests/integration/search.year.test.ts`.
</objective>

**Shared year-options helper**

Create `lib/listings/years.ts`:
- `export const YEAR_MIN = 1970;` `export const YEAR_MAX = 2027;`
- `export function yearOptions(): number[]` returning a DESCENDING list `[2027, 2026, ... 1970]` (newest first), mirroring the garage picker pattern (`Array.from({ length: YEAR_MAX - YEAR_MIN + 1 }, (_, i) => YEAR_MAX - i)`).

This is the single source the search UIs (16-06) and the listing form (16-07) import — do not duplicate the range anywhere else.
