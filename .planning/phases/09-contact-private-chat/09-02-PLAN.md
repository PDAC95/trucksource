<task type="auto">
  <name>Task 2: submitContact — the invariant-#5 spine (lib/actions/contact.ts)</name>
  <files>lib/actions/contact.ts, tests/unit/contact-actions.test.ts</files>
  <action>
`"use server"` action cloning lib/actions/comments.ts structure. Result union: `{ ok: true; threadId: number; existing?: boolean } | { ok: false; error: "unauthenticated" | "invalid" | "rate_limited" | "contacts_closed" | "not_found" }`.

Guard ORDER (research Pattern 3 — copy it verbatim, comment each step number):
1. `getClaims()` → no claims → unauthenticated (NEVER getSession).
2. `contactSchema.safeParse` re-validation.
3. Rate limit: head-count of contact_log rows by buyer in trailing 24h; >= 10 → rate_limited (constant `CONTACT_DAILY_LIMIT = 10`).
4. Listing pre-check, enumerated columns (id, status, seller_id, title): missing → not_found; status !== 'active' → contacts_closed; seller_id === buyer → invalid (can't contact yourself).
5. Existing-thread check via `getExistingThreadId` → return `{ ok: true, threadId, existing: true }` — NO new rows of any kind.
6. INSERT contact_log (buyer's form name/email/phone/message + listing/buyer/seller ids). **BLOCKING: any failure → return invalid, nothing else happens.**
7. `sendAdminContactCopy(...)` — attempted NOW, before the thread; best-effort; on success stamp `admin_emailed_at` on the row (owner-RLS update is not available — contact_log has no UPDATE policy, so stamp via the admin client inside notify.ts or pass the row id and do the stamp there; keep contact_log client-immutable).
8. INSERT message_threads with `contact_log_id`; on unique-violation (raced duplicate) re-read the existing thread id and continue to step 11 without inserting a message.
9. INSERT first messages row (sender = buyer, body = form message).
10. `sendNewMessageEmail` to the seller (throttled) — best-effort.
11. `revalidatePath('/messages')` + return `{ ok: true, threadId }`.

UNIT TEST (tests/unit/contact-actions.test.ts, clone social-actions.test.ts mock style) asserting at minimum:
- contact_log insert is called BEFORE message_threads insert (call-order assertion);
- contact_log insert failure → no thread insert, no message insert, returns error;
- existing thread short-circuits with zero inserts;
- rate-limit branch returns rate_limited without inserting;
- unauthenticated returns before any DB call.

- `markThreadRead(threadId)`: participant-scoped UPDATE of the VIEWER-side watermark column only (buyer_last_read_at or seller_last_read_at depending on role — read the thread first to know the role). This re-arms the email throttle.

- `blockUser(blockedId)` / `unblockUser(blockedId)`: owner-RLS insert/delete on user_blocks. Block stops future sends via the messages INSERT policy — no app-side message check needed (single enforcement point).

Rate-limit-hit error copy contract: actions return machine errors; UI maps to honest English ("You've reached the daily contact limit. Try again tomorrow." / "You're sending messages too fast. Wait a minute.").
