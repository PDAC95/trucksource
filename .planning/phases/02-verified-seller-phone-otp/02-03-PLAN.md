Logic, performed as close to atomic as Supabase allows (do the counts then the insert; document the small race and that the conservative caps + Twilio backstop absorb it):
  - Spend cap FIRST among the counters: count rows in otp_send_attempts where created_at > now()-24h (GLOBAL, all users). If >= `OTP_SEND_DAILY_CAP` env (default 200), return { ok:false, reason:'spend_cap' }.
  - Per-phone hour: count where phone_e164 = e164 AND created_at > now()-1h. If >= 3, return rate_limited.
  - Per-phone day: count where phone_e164 = e164 AND created_at > now()-24h. If >= 5, return rate_limited.
  - Per-IP hour: count where ip = ip AND created_at > now()-1h. If >= 3, return rate_limited. (parallel per-IP cap so phone-rotation behind one IP is throttled — PITFALL #5.)
  - Per-IP day: count where ip = ip AND created_at > now()-24h. If >= 5, return rate_limited.
  - If all pass: insert one otp_send_attempts row {user_id:userId, phone_e164:e164, ip} and return { ok:true }.
  Make OTP_SEND_DAILY_CAP read from env with a numeric default of 200, parsed safely.

**lib/actions/verify.ts** — `'use server';`. Three actions. Author sendOtp VERBATIM from 02-RESEARCH.md Pattern 1, then add the abuse logging:

`sendOtp(input: unknown)` in this EXACT order (first failure short-circuits):
  1. `const { isBot } = await checkBotId();` if isBot -> logAbuse('bot_blocked', ...) then return { ok:false, error:'blocked' }.
  2. auth: `createClient()` + `getClaims()`; userId = data?.claims?.sub; if !userId return { ok:false, error:'unauthenticated' }.
  3. validate+normalize+geo: `sendOtpSchema.safeParse(input)`; if !success return invalid. `const e164 = toE164Plus1(parsed.data.phone)`; if !e164 -> logAbuse('region_blocked', ...) return { ok:false, error:'region_unsupported' }.
  4+5. IP from `(await headers()).get('x-forwarded-for')?.split(',')[0] ?? 'unknown'`; `const gate = await consumeSendBudget({ userId, e164, ip })`; if !gate.ok: if reason==='spend_cap' -> `await alertSpendCap({userId,e164,ip})`; else logAbuse('rate_limited',...); return { ok:false, error: gate.reason }.
  6. ONLY NOW: persist the (unverified) phone to the caller's profiles_private (update phone=e164 via the cookie-bound user client so RLS owner-scopes it) so resume-on-abandon knows the number, then `await sendVerification(e164)`. return { ok:true }.

`checkOtp(input: unknown)`:
  - getClaims() -> userId (else unauthenticated).
  - checkOtpSchema.safeParse -> code (else invalid).
  - Read the caller's current phone from profiles_private (user client, owner RLS). If null -> { ok:false, error:'no_pending' }.
  - `const approved = await checkVerification(phone, code);` if !approved return { ok:false, error:'invalid_code' }.
  - On approved: update profiles_private set phone_verified_at = now() for the owner. return { ok:true }.

`acceptTerms(input: unknown)`:
  - getClaims() -> userId (else unauthenticated).
  - acceptTermsSchema.safeParse (else invalid).
  - update profiles_private set marketplace_terms_accepted_at = now(), terms_version = parsed.data.termsVersion (owner RLS). return { ok:true }.

All three use createClient() (cookie-bound, getClaims) for the user's own profiles_private writes — owner RLS applies. Only ratelimit/alert use the admin client (abuse tables). Never getSession.
