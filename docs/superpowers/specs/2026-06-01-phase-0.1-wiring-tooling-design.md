# Phase 0.1 — Wiring & Production-Ready Tooling — Design

**Date:** 2026-06-01
**Status:** Approved (design)
**Phase position:** Phase 0.1, between Phase 0 (Setup & Scaffolding, merged) and Phase 1 (Foundation & Privacy Model). Inserted work; Phases 1–10 keep their numbers.

## Goal

Connect the merged scaffold to real services and lay a production-ready quality base, so Phase 1 writes only features and migrations — never infrastructure.

## Context (current state)

- Phase 0 is merged to `master`: Next.js 16.2.6 + React 19.2.4 + TS + Tailwind v4, route groups, dormant `lib/supabase/{client,server,middleware,admin}.ts`, empty `supabase/migrations/`, `.env.example`, README.
- Supabase CLI 2.98.2 and Vercel CLI 54.4.1 are installed. `gh` is authenticated (account PDAC95, repo `PDAC95/trucksource`).
- A Supabase project **Truck Source – Staging** exists. Production is deferred (see Decision below).
- No `.env.local` yet; no real credentials wired.

## Decision: Supabase environment strategy

Use **Staging only** during early phases. A separate **Production** Supabase project is created near launch (~Phase 9–10), not now. Supabase projects are physically separate databases (no in-project env tiers). Mapping: all Vercel envs (Development, Preview, Production) point at Staging until the Production project exists; at launch, only Vercel's Production env is repointed. Versioned SQL in `supabase/migrations/` (starts Phase 1) makes that rollout reproducible.

## Decision: Pre-commit vs CI split (industry standard)

- **Pre-commit (husky + lint-staged):** fast — Prettier + ESLint on _staged files only_ (~1s). Keeps malformed code out of commits without slowing the loop.
- **CI (GitHub Actions):** the heavy gate — full `tsc --noEmit` typecheck + tests + build run on every PR. Putting the full typecheck in pre-commit would make commits slow as the project grows and push people toward `--no-verify`; CI is the right home for it.

## Decision: Supabase health-check is a terminal script

`npm run check:supabase` — a Node script that instantiates the server client and performs a trivial verified call (e.g. `auth.getClaims()` / a no-op `from()` against a system view) to prove connectivity. NOT a public `/health` route — a script leaves no public surface to remove later. It must NOT create tables (that's Phase 1).

## Scope

### In scope (5 blocks)

**Block A — Quality tooling** (no credentials needed)

- Prettier: `.prettierrc` + `.prettierignore`, integrated with ESLint (no conflicting rules; add `eslint-config-prettier` to turn off stylistic ESLint rules).
- Scripts in `package.json`: `format`, `format:check`, `typecheck` (`tsc --noEmit`), `lint:fix`.
- husky + lint-staged: `.husky/pre-commit` runs `lint-staged`; `lint-staged` config runs Prettier + ESLint on staged `*.{ts,tsx,js,jsx,json,css,md}`.

**Block B — Testing** (no credentials needed)

- Vitest configured for the Next 16 / React 19 project + one unit smoke test (e.g. `cn()` from `lib/utils.ts`).
- Playwright configured + one E2E smoke test loading `/` and asserting the "Take-Off Parts" heading.
- Scripts: `test` (vitest run), `test:watch`, `test:e2e` (playwright).
- Playwright browsers installed; `test-results/` and `playwright-report/` gitignored.

**Block C — CI (GitHub Actions)**

- `.github/workflows/ci.yml`: on pull_request (and push to master) → `npm ci` → `lint` → `typecheck` → `test` → `build`. Fails the check (blocks merge) if any step fails. E2E may run in CI or be deferred to a separate job — kept minimal but present.
- Uses Node 22 to match local.

**Block D — Supabase connection (Staging)** (needs user credentials)

- `.env.local` created from `.env.example`, filled with Staging URL + anon + service_role keys (user-provided; gitignored, never committed).
- `supabase link` to the Staging project ref (so Phase 1 can `supabase migration new` / `db pull`).
- `npm run check:supabase` script added and verified green against Staging.

**Block E — Vercel + deploy** (needs user authorization)

- `vercel link` to the project.
- Env vars set per environment via Vercel (all → Staging values for now): `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY` (the last marked sensitive / not exposed to browser).
- A first preview deploy of the scaffold succeeds and serves the placeholder home.

### Out of scope (later phases)

- Any table, RLS policy, or auth logic (Phase 1).
- The real PII contract test (Phase 1 writes it on the Vitest/Playwright harness we set up here).
- The no-GPS EXIF test (Phase 5).
- The Production Supabase project (near launch).
- Twilio, Realtime, or any feature-specific service.

## Architecture / approach

- **Execution order:** Blocks A, B, C are independent of credentials and run first (and can be reviewed without secrets). Blocks D and E require user actions (pasting secrets, authorizing Vercel) and are done interactively, last.
- **Files that change together live together:** tooling configs at repo root; CI under `.github/workflows/`; tests under `tests/` (unit) and `e2e/` (Playwright) or co-located per the testing convention chosen at implementation time, kept consistent.
- **Secret hygiene:** `.env.local` is gitignored (already covered). `SUPABASE_SERVICE_ROLE_KEY` stays server-only — the health-check script runs under Node (server context), never bundled to the client. No secret is ever printed to logs or committed.

## Error handling

- If `check:supabase` fails: report the exact error (bad URL, bad key, network), do not proceed to Vercel until green.
- If `supabase link` needs an access token, guide the user to generate one; do not hardcode it.
- If a CLI version is too old for a command, surface it and use the documented alternative.
- If the Vercel deploy fails on missing env vars, set them and redeploy — never commit them to fix it.

## Verification / success criteria (what must be TRUE)

1. `npm run format:check`, `npm run lint`, `npm run typecheck`, and `npm run test` all pass on the clean tree; `npm run build` still exits 0.
2. A commit with an intentionally mis-formatted staged file is auto-fixed (or blocked) by the pre-commit hook; a clean commit passes the hook fast.
3. The CI workflow runs on a PR and gates merge on lint + typecheck + test + build.
4. `npm run check:supabase` connects to the Staging project and returns success, without creating any table.
5. `.env.local` exists locally with Staging credentials and is NOT tracked by git; `SUPABASE_SERVICE_ROLE_KEY` is never `NEXT_PUBLIC_`-prefixed and never committed.
6. The scaffold deploys to Vercel (preview) and serves the placeholder home, with env vars configured per environment (all pointing at Staging for now).
7. No table, RLS policy, or auth logic was added.
